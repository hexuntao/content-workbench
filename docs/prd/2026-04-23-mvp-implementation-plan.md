# 内容运营工作台 MVP

## 实施计划与迭代拆解

### 1. 项目目标

在 4 个迭代内交付一个可用 MVP，打通这条主链路：

`定时采集热点 -> 选题池 -> 母稿生成 -> 口语化改写/去 AI 味 -> 三平台包装 -> 人工审核 -> 导出发布包`

### 2. 团队假设

按 1-2 名工程师 + 1 名产品/内容 owner 的现实配置设计。

### 3. 交付原则

- 每个迭代结束都要有可演示成果
- 先打通链路，再优化质量
- 先支持人工审核和导出，不做自动发布
- 优先保证“可回溯”和“状态正确”

### 4. 迭代总览

#### Iteration 1：基础底座与热点采集

目标：

- 项目跑起来
- 数据落库
- 定时抓取热点
- 形成初版选题池

交付：

- Next.js 项目初始化
- Prisma + PostgreSQL
- Trigger.dev 接入
- Source/SourceItem/TopicCluster 基础表
- RSS 与网页热点采集
- 初版去重与打分
- Dashboard/Topics 基础页

验收：

- 每天定时任务可执行
- 至少能把 RSS + 一个网页源抓下来
- Topics 页面能看到候选选题

#### Iteration 2：内容生成与改写

目标：

- 从选题生成母稿
- 建立口语化改写和去 AI 味流程

交付：

- Draft/RewriteVersion 表
- 生成母稿接口
- 标题、摘要、关键词生成
- 作者风格配置读取
- 口语化改写链路
- 去 AI 味规则引擎
- Draft 页面

验收：

- 任一选题都能生成母稿
- 至少能生成 2 个改写版本
- 审阅者可以明显区分原稿与“去 AI 味”版本

#### Iteration 3：平台包装与审核台

目标：

- 输出公众号、小红书、X 长文版本
- 建立人工审核节点

交付：

- 平台包装器
- Asset 表与素材输出
- ReviewTask 表
- Review 页面
- 退回/通过流转
- 发布包数据结构

验收：

- 三个平台都有可用内容版本
- 审核通过/退回流程跑通
- 发布前内容状态清晰

#### Iteration 4：发布准备与系统收口

目标：

- 导出发布包
- 接 X/公众号草稿能力
- 做可演示版本

交付：

- PublishPackage/PublicationRecord 表
- 导出 Markdown/HTML/图片包
- 接 `qiaomu-x-article-publisher`
- 接公众号发布准备能力
- Publish 页面
- 回填平台链接
- 基础日志与错误处理

验收：

- 三个平台都能导出可用发布材料
- 至少一个渠道能生成草稿
- 全链路可 demo

### 5. 详细任务拆解

##### 5.11 审核台

- 建 `ReviewTask`
- 审核状态流转

#### Iteration 4 任务

##### 5.12 发布准备

### 6. 页面开发顺序

页面不要并行乱做，顺序应该是：

1. `Topics`
2. `Draft`
3. `Review`
4. `Publish`
5. `Dashboard`
6. `Settings`

原因：

- `Topics -> Draft -> Review -> Publish` 是主链路
- `Dashboard` 只是入口聚合
- `Settings` 可以先用简陋表单顶着

### 7. 每个页面的完成定义

#### Topics

- 能看选题列表
- 能筛选
- 能点进去生成母稿

#### Draft

- 能看母稿
- 能触发改写
- 能切换版本
- 能进入平台包装

#### Review

- 能看最终待审稿
- 能通过/退回
- 能留下审核意见

#### Publish

- 能按渠道看最终材料
- 能导出文件
- 能看到草稿链接或失败状态

### 8. 风险与应对

#### 风险 1：热点采集质量不稳定

应对：

- RSS 先做稳定主源
- Agent-Reach 源做可选增强
- 所有采集器都要保留失败日志

#### 风险 2：去 AI 味效果不稳定

应对：

- 不做单轮“洗稿”
- 用作者语料 + 禁用表达 + 多轮重写
- 审核台保留原稿与改写稿对比

#### 风险 3：平台包装质量参差不齐

应对：

- 每个平台单独建 prompt 与结构规则
- 不追求“一份稿子直接套三平台”
- 先把标题、摘要、开头三段打磨好

#### 风险 4：草稿生成不稳定

应对：

- 草稿能力是增强，不是主路径
- 主路径永远保留导出发布包
- 失败时不阻塞主流程

### 9. 验证与测试策略

MVP 阶段至少做这几类测试：

- 采集器单元测试
- dedupe/cluster 逻辑测试
- 内容状态流转测试
- 导出包生成测试
- 审核通过/退回流程测试

人工验证清单：

- 同一热点不会重复出现 5 次
- 母稿能看，不空洞
- 去 AI 味版本明显更自然
- 三个平台都不是简单复制粘贴
- 导出包内容完整

### 10. 开发节奏建议

每个迭代按这个节奏：

1. 先敲定数据结构
2. 再做 Trigger 工作流节点
3. 再做 API
4. 最后做页面

不要反过来先做页面，否则很容易 UI 很满、工作流没打通。

### 11. 设计与技能工具的正确使用方式

#### `qiaomu-design-advisor`

用于：

- Topics 页面信息层级设计
- Review 页面交互流程设计
- Publish 页面渠道卡片设计

使用原则：

- 页面开做前先做诊断
- 不直接照搬参考站，只借鉴设计系统方向

#### `qiaomu-knowledge-site-creator`

用于：

- 借鉴内容型页面组织方式
- 借鉴参考文档写法
- 借鉴代码质量约束

不用于：

- 直接生成主业务后台

#### `qiaomu-skill-publisher`

用于：

- 当模块稳定后，把能力抽成独立 skill

建议抽出的 skill：

- `topic-hunter`
- `voice-rewriter`
- `channel-packager`
- `review-guard`

### 12. 第一版上线标准

满足以下条件即可上线内测：

- 能每天自动抓热点
- 能从选题生成母稿与三平台版本
- 能进行人工审核
- 能导出发布包
- 失败不会丢内容
- 所有关键状态可回溯

这时就已经是一个可用的“内容运营工作台”，不是演示稿了。
