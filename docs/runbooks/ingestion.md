# 热点采集 Runbook

## 1. 目的

本文档定义 MVP 阶段热点采集链路的运行、排障和人工干预方式。

适用范围：

- RSS 采集
- 网页采集
- 未来接入的 X / RSSHub / Agent-Reach 采集

## 2. 采集链路

标准链路如下：

1. 定时任务读取 `Source`
2. 根据 `Source.type` 选择采集器
3. 抓取原始内容
4. 执行 normalize
5. 写入 `SourceItem`
6. 计算 `dedupeHash`
7. 触发聚类和评分更新

## 3. 值班前检查

每次处理采集问题前先确认：

- 失败的是单个源、单个采集器，还是整个定时任务
- 最近一次成功时间
- 是否有上游网站结构变化或限流
- 是否已经出现重复入库

## 4. 正常运行基线

MVP 阶段至少应满足：

- 每日定时任务能按计划触发
- 单个源失败不会阻塞其他源
- 原始内容抓取失败时保留错误日志
- 去重失败不能导致整批任务中断

## 5. 日志要求

每次采集尝试至少记录：

- `jobId`
- `sourceId`
- `sourceType`
- `fetchStartedAt`
- `fetchFinishedAt`
- `fetchedCount`
- `insertedCount`
- `dedupedCount`
- `status`
- `errorCode`
- `errorMessage`

## 6. 常见故障与处理

### 6.1 RSS 无法拉取

现象：

- 返回超时
- 返回 `4xx` 或 `5xx`
- 内容为空

处理：

- 检查源地址是否变更
- 检查是否被上游限流
- 单独重试该源
- 若连续失败，将 `Source.status` 标记为 `ERROR` 并保留原因

### 6.2 网页结构变化导致正文抽取失败

现象：

- 标题抓到但正文为空
- 正文包含大量导航、广告、脚本内容

处理：

- 记录原始 HTML 片段或提取失败摘要
- 调整该站点的 normalize 规则
- 在修复前允许仅保存标题、链接、摘要，不阻塞整条任务

### 6.3 出现重复素材

现象：

- 同一热点短时间内反复出现在 Topics

处理：

- 检查 `sourceId + url` 唯一约束是否生效
- 检查 `dedupeHash` 计算是否包含不稳定字段
- 检查 normalize 后文本是否在不同采集器中格式差异过大

### 6.4 单源失败拖垮整批任务

现象：

- 一个源报错后，本轮后续源没有继续执行

处理：

- 确认工作流是否按源粒度隔离重试
- 要求任务实现“单源失败，批次继续”的容错
- 汇总失败结果到最终报告，而不是中途终止整个批次

## 7. 幂等策略

- 重跑同一抓取任务不能重复写入相同 `SourceItem`
- 导致重复的主要风险字段是 `url` 规范化不一致
- 对同一页面应先做 URL 标准化，再参与唯一性判断
- `dedupeHash` 应尽量基于标准化标题和正文摘要，而不是抓取时间

## 8. 人工重试原则

适合直接重试的场景：

- 网络抖动
- 临时限流
- 对象存储上传失败

不应直接重试而应先修复配置的场景：

- 采集源地址错误
- 选择器失效
- 认证信息失效

## 9. 升级规则

出现以下情况应升级为产品或工程问题，而不是只做值班处理：

- 连续两天热点池显著缩水
- 多个核心源同时失败
- Topics 明显出现大规模重复或空标题
- 上游平台协议变化导致采集方式整体失效

## 10. 事后复盘模板

每次严重采集故障结束后至少记录：

- 故障开始时间
- 影响范围
- 根因
- 临时止损动作
- 永久修复动作
- 是否需要新增监控或测试
